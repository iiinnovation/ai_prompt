# AI Prompt 版本升级计划

## 1. 文档目标

本计划用于将 [ai_prompt_update_report.html](./ai_prompt_update_report.html) 中的升级建议，整理为符合软件产品版本迭代节奏的执行方案，并对齐当前 GitHub 仓库的发布方式。

适用范围：

- 当前仓库 `ai_prompt`
- 当前远程仓库 `origin: https://github.com/iiinnovation/ai_prompt.git`
- 后续源码版本、Git Tag、GitHub Release、README 更新

---

## 2. 当前基线判断

### 2.1 当前产品基线

- 升级建议报告标注当前版本为 `v1.0.1`
- `README.md` 更新日志也记录了 `v1.0.1`
- `manifest.json` 当前版本仍为 `1.0.0`

这说明当前仓库存在“产品版本描述不一致”的问题。  
在进入下一轮功能升级前，必须先统一版本基线，否则后续 GitHub tag、release note、用户安装包版本都会混乱。

### 2.2 当前技术基线

当前项目仍属于轻量级浏览器扩展阶段，具备以下特点：

- 平台支持采用硬编码配置，新增平台成本低，但扩展性一般
- 页面注入主要依赖平台域名和选择器匹配
- 数据存储仍以 `chrome.storage.local` 为主
- 产品主体仍是“模板管理 + Prompt 注入”工具，不是“服务化 AI 产品”

因此，升级应该遵循：

1. 先修基线
2. 再做高收益、低耦合迭代
3. 最后再进入 API、社区、推荐等重能力阶段

---

## 3. 版本策略

建议采用语义化版本：

- `patch`：Bug 修复、文档修正、兼容性修复、非破坏性小优化
- `minor`：新增平台、新增模板能力、新增同步与交互能力
- `major`：权限模型变化、服务端能力引入、产品定位升级

### 版本编号建议

若 `v1.0.1` 已经实际对外发布：

- 下一版本从 `v1.0.2` 开始

若 `v1.0.1` 尚未正式发布，仅存在文档记录：

- 先将当前仓库整理并正式发布为 `v1.0.1`
- 后续功能升级从 `v1.1.0` 开始

为避免重复返工，当前建议默认采用第一种策略，即：

- 基线整理版：`v1.0.2`
- 平台扩展版：`v1.1.0`
- 数据同步版：`v1.2.0`
- 通用注入版：`v1.3.0`
- 服务化能力版：`v2.0.0`

---

## 4. 分阶段升级路线

## Phase 0：版本基线整理

### 目标版本

`v1.0.2`

### 目标

统一仓库中的版本信息，建立后续 GitHub 发布的标准动作。

### 范围

- 统一 `manifest.json` 版本号
- 更新 `README.md` 中的版本记录与仓库信息
- 补充或整理 changelog
- 确认 Git tag 规范：`vX.Y.Z`
- 形成首个可追溯的 GitHub Release 模板

### 建议产出

- 代码版本号一致
- README 更新日志一致
- Git tag 一致
- GitHub Release Notes 一致

### 验收标准

- 仓库内不再出现 `manifest` 与文档版本不一致
- 可以明确回答“当前 GitHub 对外版本到底是哪一个”

---

## Phase 1：平台扩展与模板增强

### 目标版本

`v1.1.0`

### 目标

优先做用户最容易感知、最直接提升可用范围的升级。

### 范围

#### 1. 新增平台适配

首批建议优先支持：

- Claude.ai
- Perplexity
- Grok
- 豆包

第二批可候选：

- 文心一言
- 讯飞星火
- Copilot
- Poe

#### 2. 扩充模板库

优先补充以下高频场景：

- 产品/运营：PRD、需求文档、竞品分析
- 数据分析：数据解读、报表描述、结论摘要
- 社交媒体：小红书、公众号、短内容文案
- 图像生成：Midjourney / DALL·E Prompt

### 暂不纳入

- 通用输入框适配
- 多设备同步
- AI 一键优化
- 模板社区共享

### 原因

这一阶段主要是“扩内容、扩平台”，实现边界清晰、用户价值直接、适合作为第一个明确的 minor 版本。

### 验收标准

- 至少完成 4 个新增平台适配
- 新增模板分类可在 README 中清晰说明
- 每个平台至少完成 1 次手工注入验证

---

## Phase 2：数据同步与效率提升

### 目标版本

`v1.2.0`

### 目标

解决“模板数据不跟设备走”和“高频使用效率不足”的问题。

### 范围

#### 1. 数据同步

- 将模板与设置迁移到 `chrome.storage.sync`
- 设计 `local -> sync` 的兼容迁移逻辑
- 保留本地导出能力作为兜底备份

#### 2. 使用效率增强

- 支持变量默认值，如 `{{语言|中文}}`
- 增加最近 Prompt 历史记录

### 关键约束

- `chrome.storage.sync` 存在容量与写入频率限制
- 不建议把全部历史记录都放进 sync

### 推荐方案

- 模板、分类、设置：走 `sync`
- 注入历史、临时状态：继续走 `local`

### 验收标准

- 老用户升级后原有模板不丢失
- 同账号多设备可同步核心模板数据
- 历史记录不影响同步配额

---

## Phase 3：通用注入与轻交互优化

### 目标版本

`v1.3.0`

### 目标

从“支持若干 AI 平台”升级为“尽量支持更多网页输入场景”。

### 范围

- 通用输入框适配
- 面板内联编辑

### 技术重点

当前实现依赖平台白名单与预设选择器。  
如果要支持非既有平台，需要重新设计以下能力：

- 通用 `textarea` / `contenteditable` 检测
- 焦点输入框识别策略
- 非 AI 页面上的注入回退逻辑
- 权限策略与用户信任边界

### 权限建议

优先评估以下方案：

1. 基于 `activeTab` 做按需注入
2. 尽量避免直接扩大为全站常驻权限

原因：

- 更适合 GitHub 开源项目的透明发布
- 对用户权限感知更友好
- 后续上架浏览器扩展市场时阻力更小

### 验收标准

- 非既有平台页面可识别常见输入框
- 注入失败时有明确反馈
- 不因权限升级引入明显使用门槛

---

## Phase 4：服务化能力升级

### 目标版本

`v2.0.0`

### 目标

把产品从“本地 Prompt 模板插件”升级为“具备增值能力的 Prompt 产品”。

### 范围

- Prompt 一键 AI 优化
- 页面上下文推荐
- Prompt 效果打分
- 模板社区共享

### 为什么定义为 Major

这一阶段通常会引入以下至少一项：

- 外部 API
- 服务端存储
- 用户数据管理
- 隐私说明与权限升级
- 账号体系或模板市场机制

这已经不是普通功能补充，而是产品形态变化，因此建议进入 `v2.0.0`。

### 前置条件

- 明确 API 成本与配额
- 明确隐私政策与数据边界
- 明确服务端选型与运营成本

---

## 5. GitHub 发布策略

既然当前仓库已经是通过 `git clone` 获取，后续应直接采用标准 GitHub 发布流程，不再使用“本地版本说明为主”的方式。

### 分支建议

建议采用轻量分支策略：

- `main`：稳定可发布代码
- `feature/...`：单一升级项开发
- `release/vX.Y.Z`：需要集中收口时再使用

对于当前项目体量，不建议引入过重的 Git Flow。

### Tag 与 Release 规范

每个正式版本都应同步执行：

1. 修改 `manifest.json` 版本号
2. 更新 `README.md` 更新日志
3. 提交版本收口 commit
4. 创建 tag：`vX.Y.Z`
5. 在 GitHub 创建对应 Release

### Release Notes 模板建议

每次发布至少包含：

- Added：新增能力
- Changed：行为变化
- Fixed：修复问题
- Known Issues：已知限制
- Upgrade Notes：升级说明

---

## 6. 每个版本的交付门禁

每次准备推送 GitHub 前，至少完成以下检查：

### 基础检查

- 版本号是否一致
- README 是否更新
- 是否包含清晰 changelog
- 是否残留无关临时文件

### 功能检查

- 模板搜索
- 模板注入
- 变量填写
- 覆盖/追加模式
- 排序与偏好保存
- 导入导出能力

### 平台检查

- 每个受支持平台至少完成一次手工 smoke test

### 发布检查

- commit message 符合约定
- tag 命名正确
- release 标题与版本号一致

---

## 7. 当前建议的近期执行顺序

建议接下来按以下顺序推进：

1. 先做 `v1.0.2` 基线整理
2. 再做 `v1.1.0` 平台扩展与模板增强
3. 之后做 `v1.2.0` 同步与效率优化

不建议现在就直接进入：

- 模板社区
- AI 优化
- 智能推荐

原因是这些需求会显著抬高复杂度，且当前仓库还没有完成版本基线统一和发布流程标准化。

---

## 8. 风险与注意事项

### 版本风险

- 当前 `manifest` 与文档版本不一致，若不先统一，后续 release 会失真

### 技术风险

- 平台适配依赖 DOM 结构，目标网站改版会导致失效
- `storage.sync` 配额限制可能影响同步设计
- 通用注入会触发权限与兼容性问题

### 产品风险

- 一次性塞入过多升级项，会导致测试面过大
- 轻量工具过早服务化，可能拉高维护成本

---

## 9. 结论

当前最合理的升级路线不是“按报告逐项堆功能”，而是：

- 先统一版本基线
- 再做平台与模板扩展
- 再做同步与效率优化
- 最后再进入服务化能力

这条路线更符合软件产品的版本迭代规律，也更适合后续在 GitHub 上做持续发布、版本追踪和 release 管理。
